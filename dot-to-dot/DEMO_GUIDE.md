## 📌 補強 1：物流系統 - 能力過濾驗證測試

### 目的
向評審證明：智能過濾是**真的資料庫關聯**，不是前端假過濾

### 完整測試場景

**準備工作：創建測試帳號**

1. **User A（機車騎士）**
   ```
   進入個人資料（Profile）
   → 設定 provider_profiles：
      - has_vehicle = 1  
      - vehicle_type = 'motorcycle'
      - willing_to_carry = 1
   ```

2. **User B（汽車司機）**
   ```
   設定 provider_profiles：
      - has_vehicle = 1
      - vehicle_type = 'car'
      - willing_to_carry = 1
   ```

3. **創建需要大型車輛的訂單**
   ```
   物品清單：
   - 床墊 x1
   - 書桌 x1  
   - 大箱子 x3
   
   系統自動推薦車型：🚚 貨車 or 🚐 廂型車
   ```

---

### 測試步驟

#### 場景 A：機車騎士視角（證明過濾生效）

1. 使用 **User A（機車）** 登入
2. 進入 `/logistics/lobby` 接單大廳
3. **驗證點：**
   - ❌ **看不到「需要貨車」的訂單**
   - ✅ 只看到符合機車運送的小件訂單

4. **技術驗證（重要！）**
   - 按 **F12** 打開開發者工具
   - 切換到 **Network** tab → **Fetch/XHR**
   - 查看 `GET /api/deliveries` 請求
   - 點擊查看 **Response**
   - ✅ **後端已過濾，未返回大件訂單資料**

---

#### 場景 B：汽車司機視角（證明邏輯正確）

1. 使用 **User B（汽車）** 登入
2. 進入相同的接單大廳
3. **驗證點：**
   - ✅ **看得到該大件訂單**（汽車容量足夠）
   - ✅ 訂單卡片顯示「🚗 轎車」或「🚐 廂型車」標籤

---

#### 場景 C：權限驗證（防止前端繞過）

1. **User A（機車）** 如果強制訪問該訂單詳情頁
   - 直接在瀏覽器輸入 URL：`http://localhost:5173/logistics/123`

2. 試圖點擊「接受訂單」

3. **驗證點：**
   - ❌ 後端返回錯誤：
     ```json
     {
       "error": "您的車輛類型不符合此訂單需求"
     }
     ```
   - ✅ **權限驗證正確（防止前端繞過）**

---

## 📌 補強 3：Demo 雙視窗技巧 🖥️🖥️

### 為什麼需要雙視窗？

- ❌ 切換帳號會浪費時間
- ❌ 無法展示實時通知效果
- ✅ **雙視窗可以直接並排展示毫秒級同步**

---

### 設置步驟

#### 方案 A：Chrome 一般 + 無痕模式（推薦）

```
左邊視窗（Chrome 一般模式）：
→ 登入 User A（供給者/司機/跑腿者）

右邊視窗（Chrome 無痕模式 Ctrl+Shift+N）：
→ 登入 User B（需求者/乘客）
```

**優點：**
- 不會互相干擾 cookies/localStorage
- 可以同時保持登入狀態
- 快捷鍵快速切換

---

#### 方案 B：Chrome + Edge（備用）

```
左邊：Chrome 瀏覽器 → User A
右邊：Edge 瀏覽器 → User B
```

---

### 螢幕排列建議

#### 方式 1：左右並排（推薦）

```
┌────────────┬────────────┐
│            │            │
│  User A    │   User B   │
│ (供給者)    │  (需求者)   │
│            │            │
└────────────┴────────────┘
```

**快捷鍵：**
- 左視窗：`Windows + ←`
- 右視窗：`Windows + →`

---

#### 方式 2：上下排列（備用）

```
┌────────────────────────┐
│       User A (供給者)   │
├────────────────────────┤
│       User B (需求者)   │
└────────────────────────┘
```

### 常見問題 & 解決

**Q：兩個視窗會互相登出嗎？**
- A：不會！使用無痕模式或不同瀏覽器可以同時保持登入

**Q：Socket.io 連接會衝突嗎？**
- A：不會！每個視窗都有獨立的 WebSocket 連接

**Q：如何快速定位通知？**
- A：在右邊視窗保持訂單詳情頁開啟，通知會自動更新

## 🚨 應急預案

### 如果 Socket.io 通知失敗

**症狀：** 實時通知沒有出現

**應急話術：**
> 「通知可能因為網路延遲，讓我手動刷新一下...（F5）看，資料已經更新了。在正常環境下這是毫秒級的推播。」

**快速修復：**
1. 檢查開發者工具 → Console（是否有錯誤）
2. 檢查 Network → WS（WebSocket 是否連接）
3. 重新整理頁面

---

### 如果瀏覽器視窗太小

**症狀：** 內容被擠壓，看不清楚

**應急方案：**
- 快速調整為上下排列
- 或犧牲一個視窗，用切換帳號方式演示
- 話術：「為了讓大家看得更清楚，我切換到單視窗模式...」

---

### 如果功能出現 Bug

**冷靜應對流程：**
1. 不要慌張，微笑
2. 說：「這個功能我們測試過，可能是演示環境問題」
3. 快速切換到其他功能展示
4. 最後再回來（如果時間允許）

## 🎓 評審可能的提問 & 回答準備

### Q1：Socket.io 會不會有安全問題？

**A：**
> 「我們的 Socket.io 只用於**狀態通知**，不傳輸敏感資料。真正的支付資訊走的是**零知識協議**，server 不儲存任何銀行帳號。」

---

### Q2：如果兩個司機同時接單怎麼辦？

**A：**
> 「資料庫層面有**樂觀鎖機制**（檢查 status），第一個成功的會把狀態改為 booked，第二個會收到『訂單已被接取』的錯誤。」

---

### Q3：議價流程會不會太複雜？

**A：**
> 「UI 設計上我們做了**漸進式揭露**。99% 的情況物品相符，不會看到議價表單。只有勾選『物品不符』時才展開，避免認知負擔。」

---

### Q4：實報實銷如果跑腿者亂報怎麼辦？

**A：**
> 「我們要求**上傳發票照片**（未來功能），並且有**評分機制**。亂報價的跑腿者會被差評，影響接單機會。」
